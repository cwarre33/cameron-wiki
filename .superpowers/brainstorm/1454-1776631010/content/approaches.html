Three Pipeline Architectures

Same goal, different structure. My recommendation is B. Click to select, reply in terminal.

A

Single Script

osint_scan.py — fetch → enrich → viz in one run

python osint_scan.py --protocols all --limit 500

One file, runs end-to-end. Fetches Shodan results for all 4 protocols, enriches with ASN/geo, writes D3 HTML and wiki markdown in one shot.

Pros

  • Fastest to build
  • Zero setup beyond pip install
  • Easy to share

Cons

  • Re-fetches Shodan every run (API quota)
  • Hard to re-viz without re-scanning
  • Monolithic — hard to extend
B ★

Modular Stage Pipeline

collect → enrich → graph → viz → ingest — each stage is independently re-runnable

python collect.py --protocols modbus,bacnet,dnp3,s7
python enrich.py # ASN, geo, CVE lookups
python viz.py # D3 HTML snapshot
python ingest.py # → raw/osint/ + wiki pages

Each stage reads/writes timestamped JSON in raw/osint/. Scan once, re-viz and re-ingest as many times as you want. Cached data survives across sessions.

Pros

  • Scan once, iterate on viz/wiki
  • Resumable — skip stages already done
  • Each file is focused and testable
  • Maps to wiki ingest workflow

Cons

  • 4 scripts to run in sequence
  • Need a runner script for convenience
C

Typer CLI with Subcommands

osint scan / osint enrich / osint viz / osint ingest

osint scan --protocols all --limit 1000
osint enrich --run modbus,bacnet
osint viz --protocol dnp3
osint ingest --wiki

Polished CLI tool with --help, flags, validation. Looks like a real tool. Same modular stages as B but packaged as an installable CLI.

Pros

  • Best portfolio UX
  • Installable via pip
  • Flags make it feel professional

Cons

  • Typer/Click overhead upfront
  • Packaging adds complexity before it's proven
  • Polish-before-proven is the wrong order
Recommendation: B. Modular stages match the wiki ingest pattern you already have. Start functional, promote to C-style CLI once it's proven — that's the right order given your stated priority.
